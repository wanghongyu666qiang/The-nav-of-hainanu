###  二、 逻辑模块是怎么运转的？

我们把系统切分成了四个核心逻辑层：

#### 1. 核心数据层 (Graph.h)
这是整个项目的心脏。里面定义了一个“大包裹”——结构体 `Graph`。
它包含两个极其重要的成员：
* `map<int, Scene> scenes`：用来装全校景点的名册。
* `map<int, vector<Edge>> adjList`：用来装路网的“邻接表”（记录每个点连着哪些路，且每条路都标记了是人走还是车开）。
**它是唯一的“信息汇总点”，项目中所有的操作，都是在对这个大包裹进行揉捏。**

#### 2. 黑匣子后勤层 (`curd.cpp`)
这个模块专门跟 data 夹里的 `.txt` 文件打交道。
*   **读（Load）**：程序刚启动时，把 txt 里的文本翻译成内存里的 `Graph` 结构。
*   **写（Save）**：把内存里的 `Graph` 结构格式化输出，覆盖掉旧的 txt 文件。
*   **改（CURD）**：提供了 `addScene`、`deleteRoad` 等操作，确保每次你在地图上挖掉一个点，相连的路也会被干脆利落地切断，保持数据完美一致。

#### 3. 算法计算层 (`algorithm.cpp` 待开发)
这里不关心文件长啥样，它只接客。
这里面驻扎着你的 `Dijkstra` 和 `TSP` 算法函数。当 main.cpp 把装满数据的 `Graph` 扔给它，并告诉它“我是行人，我要从 1 走到 4”。
它就会利用“以时间换空间”的法则，巧妙避开所有的机动车道，最后吐出一条经过严密计算的结点路线数组（如 `[1, 3, 5, 4]`）。

#### 4. 交互层 (main.cpp)
它是整个工程的主演点。它的运行流程非常线性：
1. 声明一个空白图：`Graph myCampus;`
2. 命令后勤部干活：`loadScenes(myCampus, ...)`
3. 开启一个巨大的 `while(true)` 循环，在黑窗（Terminal）里打印 `1. 查询最短路 2. 增加景点...` 等菜单。
4. 根据用户的输入，拿着 `myCampus` 分别去召唤 `curd` 或 algorithm 里的特定函数进行反馈。

###  三、 为什么说这套架构很“扛造”？

1.  **极度解耦**：如果你哪天觉得控制台黑框框太丑了，想上 Qt 做可视化！你的 Graph.h、`curd.cpp` 和 `algorithm.cpp` **连一个标点符号都不需要改**！你只要把 main.cpp 扔掉，换成 Qt 的窗口即可。这就叫前后端分离。
2.  **没有多重定义**：所有人都 `#include "Graph.h"`，通过头文件沟通，实现了完美的隔离和链接。
3.  **单数据流**：只有一份真实的数据，不会出现人和车的路线文件内容打架的情况。

整个系统其实非常通透。现在，图纸已经铺在桌子上了，只要把 `curd.cpp` 里的文件读写代码照着填好，把 `main` 跑动起来，这栋大楼的地基就稳稳扎住了！
